YODA Framework Voltar para a série
Post 03

Um agente com contexto zero entra num repositório

Ele não sabe quem somos, o que queremos, onde estão as regras e por que aquele arquivo estranho existe. Mesmo assim, ele chega educado e diz que entendeu.

Esse é um dos momentos mais perigosos do desenvolvimento com IA:

O agente acabou de chegar.

Não conhece o projeto.

Não conhece as manias do repositório.

Não sabe quais arquivos são atuais, quais são históricos, quais são armadilhas arqueológicas e quais existem só porque alguém, em algum dia difícil, precisou resolver um problema às pressas.

Mesmo assim, ele quer ajudar.

E agente querendo ajudar é uma coisa linda.

Até ele começar pelo arquivo errado.

O repositório tem cheiro

Para um humano, um repositório tem cheiro.

Você bate o olho e percebe algumas coisas.

O README parece ser para usuário.

A pasta docs/ talvez seja site publicado.

A pasta bootstrap-legacy/ tem cara de museu.

Um arquivo chamado REPO_INTENT.md provavelmente não está ali para decorar.

Um diretório chamado yoda/scripts/ indica que talvez exista automação e talvez seja melhor não sair editando markdown na mão como quem passa manteiga no pão.

Humano experiente faz essa leitura meio no faro.

O agente não.

O agente chega num repositório como quem entra numa cidade nova sem mapa, mas com muita disposição para dirigir.

Contexto zero não é problema. Problema é fingir que contexto zero é contexto suficiente.

A ilusão do README universal

Eu gosto de README.

README é a porta de entrada humana. É onde a gente explica o que o projeto é, como instala, como usa, qual é a idéia geral.

Mas README não resolve tudo.

Principalmente quando o leitor não é exatamente um humano curioso, mas um agente prestes a modificar arquivos.

O humano lê o README e entende a proposta.

O agente precisa de outra coisa também: instrução operacional.

Qual arquivo ler primeiro?

Qual fonte de verdade vence em caso de conflito?

Onde estão as issues?

Qual script seleciona o próximo trabalho?

Quando ele pode implementar?

Quando ele deve parar e pedir confirmação?

README responde parte disso. Mas, no YODA, essa responsabilidade começou a ganhar arquivos próprios.

A porta da frente do agente

Foi assim que a entrada do agente virou uma preocupação central.

Não bastava dizer “leia a documentação”.

Isso é quase o mesmo que dizer “seja responsável”. Bonito, mas pouco executável.

O agente precisava de uma trilha:

Leia o arquivo de intenção do repositório.

Leia o playbook do YODA.

Entenda onde estão as specs.

Use a issue atual como contrato.

Quando houver script, siga o runbook do script.

Não invente processo paralelo porque a pasta parece simpática.

Isso parece óbvio depois que está escrito.

Antes, não era.

Pequena tragédia recorrente: quando o projeto não diz por onde entrar, o agente escolhe uma porta. E nem sempre é uma porta. Às vezes é uma janela. Às vezes é um armário.

AGENTS, GEMINI, CLAUDE e a vida real

Outra parte divertida: as ferramentas não combinam entre si qual arquivo de instrução devem procurar.

Uma olha para AGENTS.md.

Outra usa GEMINI.md.

Outra pode ter outro arquivo.

E o projeto host pode já ter suas próprias instruções, que não deveriam ser sequestradas por um framework embutido.

Esse detalhe parece pequeno até você perceber o tamanho do estrago de um init que sai criando ou alterando arquivos na raiz do projeto dos outros.

Por isso o YODA foi ficando mais cuidadoso.

Ele passou a ter arquivos de entrada próprios dentro de yoda/.

O host pode apontar para eles se quiser.

Mas o YODA não precisa entrar na casa dos outros trocando a fechadura.

Essa decisão parece administrativa.

Na verdade, é uma decisão de convivência.

O agente precisa saber quando não agir

Entrada clara não serve só para começar.

Serve também para segurar.

O agente precisa saber quando deve estudar, quando deve documentar, quando pode implementar e quando precisa avaliar.

Precisa saber que YODA Flow não é uma senha mágica para sair alterando arquivo.

Precisa saber que cada fase tem autorização.

Precisa saber que a issue Markdown é fonte de verdade, mas os scripts dizem como atravessar o fluxo sem derrubar os copos.

Em outras palavras: o agente não precisa só de contexto.

Precisa de limites.

Limite é uma palavra injustiçada. Parece freio. Mas, nesse caso, é pista.

Um bom arquivo de entrada não transforma o agente em gênio. Só evita que ele comece o trabalho cavando no quintal errado.

O que ficou dessa fase

A grande lição do contexto zero é que nenhum agente deveria precisar adivinhar a topologia do projeto.

Se existe uma ordem de leitura, ela deve estar escrita.

Se existe uma fonte de verdade, ela deve estar nomeada.

Se existe um fluxo, ele deve ter uma entrada operacional.

Se existe um script que decide o próximo passo, o agente deve usar o script.

E se o agente não sabe qual desenvolvedor está operando o fluxo, ele não deveria inventar. Deveria perguntar.

Parece simples.

De novo essa frase perigosa.

Mas foi essa simplicidade que abriu a próxima porta da odisséia.

Depois de ensinar o agente a entrar no repositório, faltava transformar a issue em algo mais sério do que um lembrete bonito.

Porque se a issue continua vaga, o agente até entra pela porta certa.

Mas lá dentro ainda pode sair dançando em cima da mesa.
