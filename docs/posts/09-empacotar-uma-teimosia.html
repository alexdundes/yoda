YODA Framework Voltar para a série
Post 09

Empacotar uma teimosia

Uma coisa é funcionar no meu repositório. Outra é sair de casa sem quebrar a casa dos outros.

Até aqui, o YODA estava crescendo dentro do próprio repositório.

Isso é confortável.

Perigosamente confortável.

Porque, dentro do próprio repositório, muita coisa funciona por proximidade.

O arquivo está ali.

O script está ali.

A spec está ali.

O histórico está ali.

A bagunça também está ali, mas você já conhece a bagunça pelo nome.

É como morar sozinho: a cadeira no meio da sala não é problema, é “layout”.

Aí você decide que o framework precisa sair de casa.

E tudo muda.

Porque uma coisa é o YODA funcionar comigo olhando.

Outra é ele entrar em outro projeto e se comportar como visita educada.

O pacote é onde a fantasia acaba

Empacotar parece uma etapa administrativa.

Não é.

Empacotar é responder perguntas inconvenientes.

O que entra?

O que fica fora?

Qual é a versão?

Como eu sei de qual commit isso veio?

Como eu provo que esse arquivo não foi alterado no caminho?

Como instalo sem apagar dados do projeto host?

Como atualizo?

Como volto atrás se der ruim?

De repente, a discussão deixa de ser só processo de agente.

Vira distribuição.

E distribuição tem um talento especial para humilhar otimismo.

Empacotar o YODA foi transformar uma mania local em algo que sobrevive fora do meu colo.

O artefato precisava ser pequeno e rastreável

A primeira decisão foi dolorosamente simples:

o pacote não podia ser o repositório inteiro.

Não fazia sentido levar project/specs/ para dentro de todo projeto consumidor.

Não fazia sentido levar material de bootstrap.

Não fazia sentido levar teste, cache, arquivo temporário, poeira histórica e lembrança emocional.

O pacote precisava carregar o que o projeto host precisa para usar o YODA.

Manual embutido.

Entrypoints locais para agentes.

Scripts.

Templates.

Licença.

README curto.

Changelog.

Manifest.

Só.

O resto é bastidor.

E bastidor não deveria cair na mochila do usuário.

O package.py virou adulto responsável

O package.py virou a fonte da verdade do empacotamento.

Isso foi importante.

Porque empacotar na mão é uma forma criativa de criar versões com personalidade própria.

Uma release inclui arquivo demais.

Outra esquece um script.

Outra saiu com spec interna.

Outra nasceu com nome parecido, mas não igual.

Outra tem changelog escrito no improviso porque “depois a gente arruma”.

Depois nunca arruma.

Então o pacote virou comando.

Determinístico.

Com lista de inclusão.

Com lista de exclusão.

Com tar.gz.

Com nome versionado.

Com build metadata.

Com changelog estruturado.

Com latest.json atualizado.

Com checksum.

Ou seja: menos “eu acho que gerei certo” e mais “o comando sabe o que precisa acontecer”.

Manifest: o RG do pacote

O manifest entrou para responder uma pergunta básica:

“Que pacote é esse?”

Versão.

Build.

Commit de origem.

Data de build.

Quem construiu.

Hash do pacote.

O que entrou.

O que ficou fora.

Isso parece burocracia.

Até você precisar descobrir por que dois projetos dizem estar na mesma versão, mas um deles se comporta diferente.

Aí o manifest deixa de ser burocracia.

Vira lanterna.

latest.json e checksum: confiança sem teatro

O latest.json ficou como o cartão de visita remoto do pacote.

Ele diz a versão.

Diz o build.

Diz a URL do artefato.

Diz o sha256.

O instalador usa isso para baixar e verificar.

Não é assinatura criptográfica sofisticada.

Não tenta fingir que resolveu toda a segurança do planeta.

Mas já impede uma classe inteira de acidentes bobos.

Baixou errado?

Corrompeu?

Não bateu o hash?

Para.

Isso é uma palavra pequena e maravilhosa em software:

para.

O instalador de uma linha

Instalador de uma linha é bonito.

Também é um pouco assustador.

curl | sh é aquela coisa que todo mundo já viu, muita gente usa e quase todo mundo explica com uma tosse moral antes.

Por isso o YODA precisou tratar isso com honestidade.

Existe o caminho rápido.

E existe o caminho mais responsável.

O rápido instala a última versão pelo latest.json.

O responsável fixa a versão.

O manual baixa, confere checksum e instala com mais controle.

Não é sobre fingir que facilidade não tem risco.

É sobre deixar o risco visível e dar opção.

Pequena verdade operacional: quanto mais fácil é instalar, mais importante é deixar claro o que será baixado, verificado, copiado e preservado.

Não quebrar a casa dos outros

Essa foi a regra que mais importou.

O YODA é embutido dentro de outro projeto.

Logo, ele não é dono da casa.

Ele não deve sair alterando README.md da raiz.

Não deve tomar controle dos arquivos de agente do projeto host.

Não deve apagar issues do usuário.

Não deve tratar dados locais como se fossem parte descartável do pacote.

O pacote pode substituir framework files.

Scripts.

Templates.

Manual.

Entrypoints locais do próprio YODA.

Mas precisa preservar dados.

yoda/project/issues/.

yoda/project/extern_issues/.

Pastas legadas quando aplicável.

Esse limite mudou o tom do projeto.

O YODA deixou de ser um script esperto dentro do meu repo e passou a ser um hóspede com regras.

Update e rollback

Instalar é só o primeiro encontro.

O relacionamento começa no update.

Atualizar framework embutido é delicado porque você quer trocar o motor sem jogar fora o porta-luvas.

O update precisava verificar versão.

Baixar ou receber fonte local.

Validar checksum.

Fazer backup.

Substituir só arquivos do framework.

Preservar dados.

Rodar init para finalizar sync e migração.

E, se algo desse errado, existir uma história de volta.

Rollback não é pessimismo.

Rollback é respeito por terça-feira às 17h42.

Porque é exatamente nesse horário que uma atualização aparentemente simples decide testar sua humildade.

Release também é fluxo

O mais curioso é que empacotamento acabou virando mais um tipo de Flow.

Não com Study, Document, Implement, Evaluate no nome.

Mas com o mesmo espírito.

Leia o histórico.

Derive changelog de mudanças reais.

Gere o pacote.

Verifique o conteúdo.

Confirme que não levou o que não devia.

Confirme que levou o que precisava.

Atualize metadata.

Entregue um artefato rastreável.

Release sem esse cuidado vira superstição com número de versão.

E superstição com número de versão ainda é superstição.

O que ficou dessa fase

Empacotar o YODA trouxe uma lição simples:

um framework só começa a virar framework quando consegue sair do ambiente onde nasceu.

Antes disso, ele é uma idéia local com bons argumentos.

O pacote obrigou o projeto a separar essência de bastidor.

Obrigou a pensar em preservação.

Obrigou a pensar em integridade.

Obrigou a pensar em rollback.

Obrigou a admitir que facilidade sem cuidado é só uma forma mais rápida de criar problema.

No próximo post, a gente visita o cemitério técnico.

As coisas que pareciam boas.

As idéias que nasceram com confiança.

E os conceitos que morreram no caminho porque, felizmente, o projeto ficou mais honesto do que a primeira versão da minha cabeça.
