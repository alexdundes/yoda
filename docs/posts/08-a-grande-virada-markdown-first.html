YODA Framework Voltar para a série
Post 08

A grande virada Markdown-first

O nome dizia YAML. A realidade, teimosa como sempre, apontou para o Markdown.

YODA significa YAML-Oriented Documentation & Agents.

Bonito.

Organizado.

Com cara de arquitetura.

O tipo de nome que faz você sentir que pensou bem antes de começar.

O que, naturalmente, é uma armadilha.

Porque o projeto começou com uma fé razoável no YAML.

YAML para TODO.

YAML para log.

YAML para metadado.

YAML para dar aquela sensação de que a máquina finalmente teria uma planilha espiritual para obedecer.

E, para ser justo, YAML ajudou.

No começo.

Até começar a tentar fazer o que ele não nasceu para fazer.

YAML é ótimo até virar diário

YAML é bom para dado estruturado.

Campo.

Valor.

Lista.

Flag.

Coisa que script lê sem ficar com vontade de chorar.

Mas uma issue não é só um formulário.

Uma issue tem contexto.

Tem decisão.

Tem hesitação.

Tem “isso parece simples, mas cuidado com aquele detalhe”.

Tem a frase atravessada que explica por que uma alternativa foi descartada.

Tem o medo pequeno que não cabe em boolean.

YAML não gosta disso.

Ele até aceita.

Mas aceita com aquela cara de quem está segurando um móvel no elevador.

Você começa a escrever texto longo dentro de campo.

Depois quebra linha.

Depois indenta.

Depois descobre que um caractere inocente virou incidente diplomático.

E aí percebe:

talvez eu esteja usando o armário como sala de estar.

A virada Markdown-first foi o YODA admitindo: meu nome é YAML, mas meu coração é Markdown.

A duplicação começou a incomodar

No modelo inicial, havia uma tentação de manter estado em mais de um lugar.

Um TODO YAML dizia uma coisa.

A issue Markdown dizia outra.

O log YAML guardava histórico.

O markdown tentava contar a história.

O agente precisava juntar as peças antes de entender o que estava acontecendo.

Isso funciona por um tempo.

Como muita coisa funciona por um tempo em software.

Esse é o problema.

Funciona o suficiente para você confiar.

Depois quebra no pior momento, com uma divergência pequena e educada.

Um status aqui.

Uma fase ali.

Um log que ficou no arquivo antigo.

Uma dependência que está certa em um lugar e ausente no outro.

Pronto.

Agora você não tem fonte de verdade.

Tem fonte de debate.

A issue Markdown tomou o centro

A virada foi simples de explicar e trabalhosa de assumir:

a issue Markdown virou a fonte operacional de verdade.

Não só um texto bonitinho para humano ler.

Não só uma camada narrativa em volta do YAML.

A issue passou a carregar o contrato.

Metadados no front matter.

Contexto no corpo.

Critérios de aceite no corpo.

Notas de implementação no corpo.

Resultado no corpo.

Flow log no corpo.

Um arquivo só contando a história inteira.

Isso parece menos “arquitetural” do que espalhar dados em vários artefatos.

Mas é muito mais humano.

E, curiosamente, muito mais operacional.

Porque o agente também se beneficia quando a história está no mesmo lugar do estado.

Front matter mínimo

O YAML não morreu.

Ele só parou de querer ser protagonista.

O front matter continuou com o que precisa ser estrutura.

schema_version.

status.

phase, quando existe.

depends_on.

title.

description.

priority.

Algumas datas.

Alguma referência externa, quando existe.

Só.

YAML ficou como etiqueta.

Markdown ficou como documento.

Essa separação reduziu atrito.

O script lê o que precisa ler.

O humano lê o que precisa entender.

E ninguém precisa fingir que uma lista de campos consegue carregar nuance sozinha.

ID pelo nome do arquivo

Outra decisão pequena, mas libertadora:

o ID passou a ser derivado do nome do arquivo.

Não do front matter.

Isso parece detalhe.

Mas detalhe é onde mora metade do cansaço operacional.

Se o arquivo se chama yoda-0049-alguma-coisa.md, o ID é yoda-0049.

Fim.

Não precisa duplicar.

Não precisa reconciliar.

Não precisa abrir uma investigação porque o filename diz uma coisa e o front matter diz outra.

O arquivo virou identidade.

O resto acompanha.

Flow log dentro da issue

O log também mudou de casa.

Antes, log separado parecia limpo.

Bonito até.

Uma coisa para estado.

Outra para histórico.

Outra para texto.

Mas, na prática, a pergunta sempre era:

“O que aconteceu com essa issue?”

E a melhor resposta não deveria exigir caça ao tesouro.

Por isso o ## Flow log foi para dentro do Markdown da issue.

Transição de fase.

Ajuste manual.

Resumo de implementação.

Avaliação.

Tudo ali.

No mesmo arquivo.

Versionado junto.

Revisável junto.

Com o contexto logo acima.

Parece óbvio depois.

Quase tudo que presta parece óbvio depois.

Regra que sobrou: YAML para máquina. Markdown para história. Front matter para o mínimo. Corpo da issue para o que alguém vai precisar entender no futuro.

A migração 0.3.0

A transição para o modelo 0.3.0 não foi só filosófica.

Virou migração.

TODO YAML legado precisou virar metadado em issue Markdown.

Logs YAML precisaram ser consolidados no ## Flow log.

Scripts precisaram parar de depender do TODO YAML como índice principal.

O leitor determinístico passou a operar a partir de yoda/project/issues/*.md.

Com filtro por dev.

Com ID derivado do filename.

Com validação de schema.

Com erro claro quando alguma coisa estava errada.

Foi a hora em que a idéia deixou de ser preferência estética.

Virou contrato operacional.

O que ficou dessa virada

A grande virada Markdown-first ensinou uma coisa importante para o YODA:

o formato certo não é o mais elegante no diagrama.

É o que reduz adivinhação no uso real.

Markdown venceu porque ele é legível.

Porque aceita contexto.

Porque carrega decisão.

Porque conversa bem com git.

Porque o humano realmente lê.

E porque o agente consegue operar melhor quando não precisa montar a história a partir de pedaços espalhados.

YAML continuou útil.

Só perdeu a mania de querer narrar a novela.

No próximo post, a história sai do fluxo interno e vai para outro tipo de dor:

empacotar uma teimosia.

Porque uma coisa é criar um framework no próprio repositório.

Outra é fazer ele caber num pacote que alguém consiga instalar sem invocar espíritos antigos do terminal.
