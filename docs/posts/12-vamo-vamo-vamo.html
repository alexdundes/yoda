YODA Framework Voltar para a série
Post 12

Vamo, vamo, vamo

O fechamento da primeira temporada: eu só queria que o agente lesse o markdown. Naturalmente, terminei com framework, pacote, instalador e uma pequena coleção de cicatrizes técnicas.

Chegamos ao fim da primeira temporada.

E eu gosto da palavra temporada porque ela carrega uma promessa elegante:

isso aqui teve começo, meio e fim.

Mentira controlada.

Teve começo.

Teve meio.

O fim é mais uma placa escrita “pare aqui antes de inventar mais coisa”.

O YODA começou com uma frase pequena.

Eu só queria que o agente lesse o markdown.

Era uma vontade honesta.

Quase humilde.

Daquelas que você fala em voz alta e acredita que vai resolver com dois arquivos, uma convenção e talvez um README bem-intencionado.

Depois você pisca.

Tem issue com front matter.

Tem script decidindo fase.

Tem Intake.

Tem Flow.

Tem Prep Flow.

Tem pacote.

Tem install.

Tem update.

Tem log.

Tem schema.

Tem changelog.

E tem você olhando para a tela com aquela cara clássica de:

“como foi que eu vim parar aqui?”

A odisséia do YODA foi descobrir que pedir para o agente “seguir o contexto” não é processo. É desejo.

A primeira lição: contexto não é decoração

No começo, eu tratava contexto como uma coisa que o agente deveria buscar com boa vontade.

O markdown estava ali.

As decisões estavam ali.

A intenção estava ali.

O agente só precisava ler.

Olha que frase perigosa:

“só precisava”.

Contexto solto não é contexto operacional.

É material de consulta.

Para virar processo, ele precisa ter lugar.

Precisa ter ordem.

Precisa ter regra de entrada.

Precisa ter uma forma de dizer:

antes de tocar no código, leia isto.

Antes de avançar, registre aquilo.

Antes de concluir, prove que concluiu.

Foi aí que o YODA deixou de ser “documentação para humanos” e começou a virar “documentação que governa trabalho”.

A segunda lição: agente bom também precisa de trilho

Existe uma tentação muito forte de achar que um agente melhor resolve tudo.

Modelo melhor.

Prompt melhor.

Janela maior.

Mais ferramenta.

Mais autonomia.

Tudo isso ajuda.

Mas não substitui contrato.

O agente pode ser ótimo e ainda assim fazer uma pergunta errada.

Pode ser ótimo e escolher uma issue que não era para hoje.

Pode ser ótimo e implementar antes de documentar.

Pode ser ótimo e confundir intenção com autorização.

O YODA nasceu dessa desconfiança saudável.

Não uma desconfiança cínica.

Uma desconfiança de engenharia.

Aquela que olha para um sistema e pergunta:

onde isso quebra quando eu estiver cansado?

Onde isso quebra quando o contexto estiver grande?

Onde isso quebra quando o agente estiver confiante demais?

Foi por isso que o Flow ficou dividido em fases.

Study.

Document.

Implement.

Evaluate.

Uma etapa por vez.

Uma autorização por vez.

Um rastro por vez.

A terceira lição: script é memória com dentes

Eu poderia ter deixado tudo no prompt.

Seria mais rápido.

Também seria pior.

Prompt é ótimo para intenção.

Mas script é melhor para regra.

Script não acorda inspirado.

Script não interpreta “talvez” como “claro, pode seguir”.

Script não esquece que uma issue bloqueada não deveria ser selecionada.

Script não acha que `to-do` com `flow_prepared_until: document` é a mesma coisa que `doing/document`.

Script é chato.

E por isso mesmo é útil.

O YODA foi ficando interessante quando as decisões importantes saíram da boa vontade do agente e foram para ferramentas pequenas.

yoda_flow_next.py não escreve poesia.

Ainda bem.

Ele escolhe a próxima fase.

yoda_intake.py não tenta parecer esperto.

Ele organiza entrada.

yoda_prep_flow.py não negocia com a ansiedade.

Ele prepara sem implementar.

Essa foi uma das maiores viradas do projeto:

o agente interpreta, mas o script arbitra.

A quarta lição: processo bom remove coisas

A série também teve seus pequenos funerais.

O lightweight saiu.

tags saíram.

agent saiu.

Entry points duplicados saíram.

Defaults em template saíram.

O projeto ficou menor em alguns lugares para poder crescer em outros.

Isso parece óbvio depois.

Na hora, não é.

Na hora, cada conceito removido parece um pedaço do seu próprio raciocínio indo embora.

Mas framework que só acumula vira museu de intenção.

E museu de intenção não executa issue.

O YODA ficou melhor quando começou a perguntar:

isso ajuda o agente a trabalhar melhor?

Isso ajuda o humano a revisar melhor?

Isso reduz ambiguidade?

Isso preserva memória?

Se a resposta era não, a feature tinha que se explicar.

Algumas não conseguiram.

A quinta lição: pacote muda o pacto

Enquanto o YODA vivia só dentro do próprio repositório, ele podia ser meio informal.

Um script aqui.

Uma convenção ali.

Uma gambiarra com cara de fase transitória.

A partir do momento em que virou pacote, a conversa mudou.

Pacote precisa instalar.

Precisa atualizar.

Precisa saber versão.

Precisa ter manifest.

Precisa ter changelog.

Precisa sobreviver fora da casa onde nasceu.

Foi aí que a teimosia virou produto interno.

Não no sentido glamouroso.

No sentido mais honesto:

se outra pasta for usar isso, a outra pasta não pode depender do meu humor do dia.

Essa frase resume muito do YODA.

Processo existe para o projeto não depender do humor do dia.

O que o YODA é agora

Depois de toda essa volta, eu não diria que o YODA é um framework para “fazer IA programar”.

Isso seria pequeno demais.

O YODA é uma tentativa de organizar a relação entre humano, agente e repositório.

O humano traz intenção.

O repositório guarda memória.

O agente executa trabalho.

O script segura o contrato.

Quando uma dessas partes tenta fazer o trabalho da outra, começa a confusão.

Humano tentando lembrar tudo de cabeça.

Agente tentando adivinhar prioridade.

Markdown virando enfeite.

Script virando sugestão.

Issue virando lembrete solto.

O YODA tenta colocar cada coisa no seu lugar.

Não perfeitamente.

Mas melhor do que “vai lá e vê o que dá”.

Resumo brutal: YODA é vergonha na cara operacionalizada em Markdown, scripts e pequenas travas contra a pressa.

O que vem depois

Essa primeira temporada foi sobre criar o framework.

A próxima pergunta é mais incômoda.

Como isso se comporta em projetos reais?

Com prazos reais.

Com dívida técnica real.

Com issue mal escrita.

Com humano cansado.

Com agente querendo ajudar demais.

Com projeto que não nasceu para ser laboratório.

É aí que processo mostra se é ferramenta ou fantasia.

Eu não acho que o YODA resolve tudo.

Aliás, se um framework prometer resolver tudo, esconda sua carteira e seus finais de semana.

Mas eu acho que ele resolve uma coisa importante:

ele diminui a área onde a IA precisa adivinhar.

E isso, hoje, já é muita coisa.

Vamo, vamo, vamo

No fim, a frase ficou mais engraçada do que planejada.

Vamo, vamo, vamo.

Não como pressa.

Como ritmo.

Vamo estudar antes.

Vamo documentar direito.

Vamo implementar dentro do combinado.

Vamo avaliar sem fingir que deu certo.

Vamo remover o que atrapalha.

Vamo empacotar o que precisa sair de casa.

Vamo aceitar que agente bom não elimina processo.

Ele torna processo bom ainda mais valioso.

Era só para o agente ler o markdown.

Virou uma odisséia.

Como quase tudo que começa com “vai ser rápido”.
