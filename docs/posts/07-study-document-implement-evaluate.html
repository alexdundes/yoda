YODA Framework Voltar para a série
Post 07

Study, Document, Implement, Evaluate

Quatro fases para fazer o agente desacelerar o suficiente para entender o que está fazendo.

O fluxo parece simples.

Study.

Document.

Implement.

Evaluate.

Quatro palavras.

Uma linha.

Uma daquelas coisas que, olhando de longe, dá vontade de falar:

“Nossa, precisava mesmo de um framework para isso?”

Precisava.

Infelizmente.

Porque o problema nunca foi saber que seria bom estudar antes de codar.

Todo mundo sabe.

O problema é fazer isso quando o agente está animado, o humano está com pressa e a issue parece clara o suficiente para começar.

Parece.

Essa palavra já deveria vir com sirene.

A tentação de pular para o código

A IA tem uma qualidade que também é um defeito.

Ela quer ajudar.

Rápido.

Muito rápido.

Você descreve uma dor e, antes de terminar a frase, ela já está mentalmente abrindo três arquivos, criando uma função, mexendo no CSS e sugerindo um teste que talvez nem rode.

É sedutor.

Dá sensação de produtividade.

Dá aquela ilusão gostosa de que o projeto está andando porque a tela está mudando.

Mas movimento não é progresso.

Às vezes é só um erro com boa velocidade.

O YODA Flow nasceu para colocar uma catraca nesse impulso.

Não uma catraca burocrática de repartição pública.

Uma catraca de segurança.

Daquelas que impedem você de entrar na obra sem capacete enquanto fala “é rapidinho”.

O Flow é a placa de “devagar” na descida. Ninguém gosta dela até perceber o tamanho da curva.

Study: antes de mexer, entenda

Study é a fase mais humilhante.

Ela obriga o agente a admitir que ainda não sabe o suficiente.

E, pior, obriga o humano a admitir que talvez a issue também não esteja tão clara assim.

Nessa fase, o trabalho não é implementar.

Não é “só ajustar uma coisinha”.

Não é aproveitar que já abriu o arquivo e corrigir um negócio ali.

Study é leitura.

É mapa.

É entender onde a mudança entra, o que ela toca, que contrato existe, que teste protege aquilo e que parte da história ainda está escondida.

É o agente dizendo:

“Acho que entendi. Mas antes de sair cortando fio, deixa eu ver para onde ele vai.”

Isso parece lento.

Até você comparar com o tempo de desfazer uma implementação plausível que nasceu em cima de uma leitura errada.

Document: transforme entendimento em contrato

Depois vem Document.

Essa é a fase onde o entendimento deixa de morar na cabeça do agente e passa para o papel.

Ou melhor: para o markdown.

É aqui que a issue é refinada.

Escopo.

Fora de escopo.

Critérios de aceite.

Decisões.

Riscos.

Pontos de validação.

Document é onde a conversa vira contrato executável.

Não executável no sentido “rodar como código”.

Executável no sentido “dá para alguém seguir sem precisar adivinhar o espírito da reunião”.

Essa fase também tem um efeito colateral ótimo.

Ela expõe mentira cedo.

Se você não consegue documentar o que vai fazer, talvez ainda não saiba o que vai fazer.

Dói um pouco.

Mas dói barato.

Implement: agora sim, mas só agora

Implement é a fase que todo mundo queria desde o começo.

É onde o código muda.

É onde a coisa finalmente anda.

Mas no YODA ela chega depois de Study e Document por um motivo simples:

implementação sem contrato é chute com autocomplete.

Pode acertar.

Inclusive muitas vezes acerta.

Esse é o perigo.

Quando dá certo de vez em quando, a pressa ganha confiança.

E aí você começa a tratar sorte como método.

No Flow, Implement tem limite.

O agente implementa apenas o escopo aprovado.

Não pega carona em arquivo aberto.

Não corrige dívida lateral porque “já que estamos aqui”.

Não transforma uma issue pequena numa reforma do apartamento inteiro.

Implement é execução.

Não é turismo arquitetural.

Regra de sanidade: se apareceu uma idéia ótima no meio da implementação, ela provavelmente merece uma nova issue. Não uma invasão silenciosa no escopo atual.

Evaluate: entregar não é declarar vitória

Evaluate é a fase que salva a dignidade do commit.

É aqui que o agente precisa parar e conferir.

Os critérios de aceite foram cumpridos?

Os testes relevantes rodaram?

O comportamento esperado está mesmo lá?

Alguma coisa ficou pendente?

O resultado foi registrado?

Sem Evaluate, o fim do trabalho vira uma frase otimista.

“Pronto.”

Pronto segundo quem?

Com base em quê?

Validado onde?

Evaluate transforma o “acho que foi” em evidência.

E é por isso que o YODA exige preencher o Result log no final.

Não para enfeitar a issue.

Para deixar rastro.

Porque daqui a duas semanas ninguém lembra se aquilo foi testado, se ficou pendência, se tinha issue externa, se a decisão foi consciente ou se todo mundo só estava cansado.

O log lembra.

Uma chamada por fase

O detalhe que parece pequeno, mas muda tudo, é este:

o Flow avança uma fase por vez.

O comando yoda_flow_next.py não é uma esteira desgovernada.

Ele resolve o próximo passo determinístico, aplica uma transição válida e devolve a linha de runbook daquela fase.

Depois disso, o agente trabalha.

E espera.

Espera autorização humana explícita para avançar.

Isso é irritante no começo.

Principalmente quando você está no modo “vamo, vamo, vamo”.

Mas é justamente aí que a regra importa.

O humano continua no circuito.

O agente não se promove sozinho.

E o projeto ganha uma trilha de decisão em vez de uma sequência de palpites bem escritos.

O que ficou dessa fase

O YODA Flow trouxe uma idéia muito simples para o centro do framework:

não basta ter uma issue boa.

Também é preciso ter um rito de execução bom.

Study evita a pressa ignorante.

Document evita a memória seletiva.

Implement evita a expansão silenciosa de escopo.

Evaluate evita a vitória imaginária.

Quatro fases.

Uma por vez.

Com o humano segurando a chave da próxima porta.

No próximo post, a história dá uma virada importante.

Porque, no começo, o YODA parecia muito YAML-oriented.

Até ficar claro que YAML demais é péssimo para contar história.

A grande virada foi aceitar o óbvio que eu demorei para admitir:

markdown first.
