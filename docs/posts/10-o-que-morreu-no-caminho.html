YODA Framework Voltar para a série
Post 10

O que morreu no caminho

Nem toda idéia ruim nasce feia. Algumas nascem elegantes, ganham nome e depois atrapalham com muita convicção.

Criar conceito é gostoso.

Dá sensação de avanço.

Você nomeia uma coisa e, por alguns minutos, parece que resolveu um problema.

Às vezes resolveu mesmo.

Às vezes só colocou um crachá no problema e deu uma cadeira para ele na reunião.

O YODA teve vários desses momentos.

Campos que pareciam úteis.

Modos que pareciam elegantes.

Metadata que parecia prudente.

Automação que parecia onboarding.

Depois o projeto cresceu.

E algumas idéias começaram a fazer barulho.

Não barulho grande.

Barulho de atrito.

Aquele ruído pequeno que você ignora até perceber que metade da complexidade mora nele.

A maturidade do YODA não veio só do que entrou. Veio muito do que saiu.

Remover parece derrota

Remover conceito é mais difícil que criar.

Criar parece construção.

Remover parece admitir erro.

E ninguém gosta de olhar para uma issue antiga e pensar:

“Olha que bela complicação eu inventei com convicção.”

Mas a verdade é que remover nem sempre é derrota.

Muitas vezes é o projeto ficando mais honesto.

Ele para de carregar coisas só porque elas já existiam.

Para de sustentar abstração que não paga aluguel.

Para de exigir que o agente entenda distinções que quase ninguém usa.

Foi assim que alguns conceitos começaram a sair.

O fim do lightweight

O lightweight nasceu com uma intenção boa.

Ter um caminho mais leve.

Uma variação menor.

Um modo menos cerimonial.

Quem nunca quis isso?

O problema é que um modo alternativo não é gratuito.

Ele precisa aparecer no schema.

Nos scripts.

Nos templates.

Nos testes.

Na cabeça do agente.

Na cabeça do humano.

E, pior, ele cria uma pergunta que nunca ajuda:

“Isso aqui é flow normal ou lightweight?”

No fim, o uso real não justificava a variação.

O YODA precisava de um fluxo claro.

Não de um menu degustação de processo.

Então o lightweight saiu.

E o projeto respirou melhor.

Tags: uma gaveta que parecia útil

tags também pareciam uma boa idéia.

Tags sempre parecem.

Elas prometem organização sem compromisso.

Você coloca uma etiqueta e sente que ganhou controle.

Até perceber que ninguém sabe qual etiqueta usar.

Ou que duas etiquetas significam quase a mesma coisa.

Ou que o filtro existe, mas ninguém filtra.

Ou que o schema ficou maior para sustentar uma gaveta vazia.

No YODA, tags não estavam gerando valor proporcional à complexidade adicionada.

Então foram removidas do modelo ativo.

Sem criar outro sistema de classificação no lugar.

Essa parte é importante.

Às vezes a melhor substituição para uma feature é silêncio.

O campo agent

O campo agent parecia razoável.

Afinal, estamos falando de agentes.

Por que não registrar qual agente está associado à issue?

Porque, na prática, aquilo adicionava ruído ao modelo operacional.

O YODA não precisava saber isso para selecionar issue.

Não precisava disso para avançar fase.

Não precisava disso para validar aceite.

E manter um campo só porque ele parece semanticamente bonito é uma forma educada de acumular bagagem.

O campo saiu do schema.

As flags saíram da CLI.

Os logs antigos ficaram como história.

O presente ficou mais simples.

Entry points fora do YAML

Entry points são úteis.

Mas o lugar deles importava.

No começo, havia duplicidade.

Entry points no YAML.

Entry points no corpo Markdown.

Parece inofensivo.

Até um lugar divergir do outro.

Aí você não tem documentação.

Tem duas testemunhas discordando.

A decisão foi simples:

entry points ficam no Markdown.

Como lista legível.

Onde o humano e o agente leem o contexto.

O YAML não precisava carregar isso.

Menos duplicação.

Menos divergência.

Menos reconciliação depois.

Defaults demais no template

Template com default demais é um tipo de mentira confortável.

Ele parece pronto.

Mas parte daquela prontidão é só preenchimento automático fingindo intenção.

No YODA, o template de issue chegou a carregar metadata operacional demais.

Com isso, a responsabilidade ficava espalhada.

O template define?

O script define?

O update preserva?

O init reconcilia?

Quando muita gente manda no mesmo campo, o campo vira condomínio.

E condomínio tem reunião.

A solução foi tirar defaults operacionais do template.

Template virou estrutura.

Scripts viraram fonte de preenchimento.

Front matter ganhou ordem canônica.

Diff ficou menos barulhento.

A origem dos dados ficou mais clara.

ID no front matter

O ID no front matter também teve seu momento.

Até ficar redundante.

Se o arquivo já se chama yoda-0057-alguma-coisa.md, o ID está no nome.

Duplicar isso no front matter cria uma chance de divergência.

E divergência de identidade é uma daquelas pequenas tragédias administrativas que ninguém quer investigar.

Então o contrato ficou mais direto:

ID derivado do filename.

Front matter sem id como fonte para novos artefatos.

É um detalhe pequeno.

Mas o YODA é feito desses detalhes pequenos que impedem o processo de virar um sudoku de metadados.

Dependencies no corpo

Dependência também teve duplicação.

depends_on no front matter.

## Dependencies no corpo.

De novo: parece inofensivo.

Até um ficar atualizado e o outro não.

Dependência é dado operacional.

Script precisa ler.

Seleção determinística depende disso.

Então a fonte ficou no front matter.

O corpo não precisa repetir.

Se precisar explicar uma decisão sobre dependência, isso pode entrar no contexto ou nas notas.

Mas a lista operacional mora em um lugar só.

Init mexendo na casa dos outros

Essa foi uma remoção mais recente e importante.

O init.py já foi mais intrusivo.

Ele mexia em arquivos de agente e intent na raiz do projeto host.

Isso parecia útil para onboarding.

O projeto instalava o YODA e já ganhava instruções.

Conveniente.

Até você lembrar que o projeto host tem dono.

Ele pode ter seu próprio AGENTS.md.

Seu próprio GEMINI.md.

Seu próprio CLAUDE.md.

Seu próprio contexto.

O YODA não deveria entrar e reorganizar a geladeira.

A mudança foi deixar as instruções autocontidas dentro de yoda/.

yoda/AGENTS.md.

yoda/GEMINI.md.

yoda/CLAUDE.md.

yoda/yoda.md.

O host aponta para elas se quiser.

Mas o YODA não invade a raiz.

Regra que ficou: framework embutido precisa saber onde termina. A pasta yoda/ é território dele; a raiz do projeto host não é.

O que essas mortes ensinaram

O padrão se repetiu várias vezes.

Uma idéia parecia útil.

Entrava no schema.

Ganhava flag.

Ganhava teste.

Ganhava documentação.

Depois o uso real mostrava que ela não pagava a própria manutenção.

O YODA foi ficando melhor quando parou de confundir completude com quantidade de campos.

Menos metadata.

Menos duplicação.

Menos modos alternativos.

Menos invasão no projeto host.

Mais fonte única.

Mais Markdown onde a história importa.

Mais script onde regra precisa ser chata.

Mais humildade no contrato.

Foi aqui que eu comecei a gostar mais das remoções do que das adições.

Porque adicionar mostra imaginação.

Remover mostra critério.

No próximo post, a história muda um pouco de direção.

Depois de remover, limpar e estabilizar, apareceu uma necessidade nova:

preparar antes de executar.

Ou, em termos menos elegantes:

às vezes a melhor coisa que o agente pode fazer hoje é não implementar nada.
