Quando a issue deixa de ser lembrete e vira contrato

Uma issue ruim é uma lembrança. Uma issue boa é um combinado. Uma issue YODA tenta ser aquele amigo chato que pergunta: “mas o que exatamente você quer dizer com isso?”.

No começo, issue parecia só um lugar para escrever a tarefa. Título, duas frases, talvez um acceptance criteria para parecer adulto. Funcionava até o agente encontrar uma brecha.

A brecha é sempre elegante. Ele implementa algo plausível. Não necessariamente o que você queria, mas algo defensável. E aí começa a discussão filosófica que ninguém queria ter numa terça-feira.

O YODA empurrou a issue para outro lugar: contexto, objetivo, escopo, fora de escopo, critérios de aceite, dependências, riscos, logs. Não para burocratizar. Para evitar telepatia.

A issue vira o tênis amarrado antes da corrida. Dá para correr sem isso? Dá. Mas uma hora você tropeça.

Esse post é sobre a transformação da issue Markdown em contrato operacional. O lugar onde a história é contada, os limites são marcados e o agente perde a desculpa de “achei que era isso”.

O que este post vai cobrir

  • Por que Markdown virou a camada narrativa do YODA.
  • Como acceptance criteria mudam a conversa.
  • Por que logs dentro da issue ajudam na memória do projeto.
