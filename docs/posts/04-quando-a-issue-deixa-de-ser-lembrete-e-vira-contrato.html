YODA Framework Voltar para a série
Post 04

Quando a issue deixa de ser lembrete e vira contrato

Uma issue ruim é uma lembrança. Uma issue boa é um combinado. Uma issue YODA tenta ser aquele amigo chato que pergunta: “mas o que exatamente você quer dizer com isso?”.

No começo, issue parecia só um lugar para escrever a tarefa.

Título.

Duas frases.

Talvez um acceptance criteria para parecer adulto.

Funcionava.

Até o agente encontrar uma brecha.

A brecha é sempre elegante. Ele implementa algo plausível. Não necessariamente o que você queria, mas algo defensável.

E aí começa a discussão filosófica que ninguém queria ter numa terça-feira.

“Mas eu achei que era isso.”

Pois é.

Achou.

O problema da telepatia

Muita issue nasce esperando telepatia.

Ela diz “ajustar fluxo de intake”.

Ou “corrigir comportamento do script”.

Ou “melhorar template”.

Para um humano que estava na conversa original, talvez isso baste.

Talvez.

Para um agente chegando com contexto parcial, isso é um convite para preencher lacunas.

E agente preenchendo lacuna é como autocomplete em porta de avião: pode até acertar, mas você fica torcendo para ninguém testar em produção.

A issue precisa matar a telepatia.

Não porque o agente seja incapaz.

Mas porque a capacidade dele precisa ser apontada para uma direção verificável.

A issue vira o tênis amarrado antes da corrida. Dá para correr sem isso? Dá. Mas uma hora você tropeça.

Contrato não é burocracia

Eu sei.

Quando a gente começa a falar em contrato, escopo, fora de escopo, critério de aceite, dependência e risco, dá aquela coceira.

Parece reunião.

Parece documento corporativo.

Parece alguém transformando uma tarefa simples num ritual com vela, formulário e ata.

Mas aqui o contrato não é burocracia.

É proteção contra adivinhação.

Contexto diz de onde veio o problema.

Objetivo diz onde queremos chegar.

Escopo diz o que pode ser tocado.

Fora de escopo diz o que não deve ser tocado mesmo que pareça uma boa idéia.

Acceptance criteria dizem como saber que acabou.

Riscos dizem onde a tarefa pode morder.

Dependências dizem o que precisa estar resolvido antes.

Isso não é enfeite.

É o mapa.

Markdown virou a sala da conversa

No YODA, a issue Markdown virou o lugar onde a conversa fica.

Não só a ordem seca.

A conversa.

Por que isso importa.

O que já foi decidido.

O que não deve ser confundido.

Quais arquivos são entrada provável.

Quais critérios precisam ser validados no Evaluate.

YAML continua útil, mas para metadado.

Status.

Prioridade.

Dependências.

Coisas que script precisa ler sem drama.

Mas a história da tarefa mora melhor em Markdown.

Porque humano lê Markdown.

Agente também.

E, quando os dois precisam se entender, legibilidade deixa de ser luxo.

A virada mental: a issue não é um bilhete para o agente. É o acordo entre humano, agente e projeto sobre o que será feito.

O corpo da issue como trilha

Uma boa issue YODA não tenta ser romance.

Mas também não tenta ser telegrama.

Ela precisa conter o suficiente para orientar o trabalho sem obrigar o agente a arqueologia infinita.

Um resumo claro.

Contexto.

Objetivo.

Escopo.

Fora de escopo.

Critérios de aceite testáveis.

Riscos e casos chatos.

Notas de implementação quando ajudam.

Logs para registrar o que aconteceu.

Sem isso, o agente até pode acertar.

Mas, se acertar, fica difícil saber se acertou por contrato ou por sorte.

Acceptance criteria mudam a conversa

Acceptance criteria são uma das partes mais subestimadas da história.

Sem critério de aceite, a pergunta final vira:

“Ficou bom?”

Essa pergunta é perigosa.

Bom para quem?

Comparado com o quê?

Validado como?

Com critério de aceite, a pergunta muda:

“O contrato foi cumprido?”

Aí a conversa fica menos mística.

O Evaluate deixa de ser um aplauso final e vira conferência.

O agente não entrega só um diff. Entrega uma resposta para critérios que estavam escritos antes.

Isso muda tudo.

Logs são memória, não decoração

Outra peça importante: log.

Durante muito tempo, log pareceu acessório.

Algo que a gente preenche quando sobra paciência.

No YODA, o log virou parte da memória operacional.

Quando uma issue avança de fase, isso precisa deixar rastro.

Quando algo bloqueia, isso precisa ficar visível.

Quando o resultado é avaliado, isso precisa ser registrado de um jeito que o próximo agente consiga entender sem sessão espírita.

O log não existe para decorar a issue.

Existe para impedir que o projeto dependa da memória da última conversa.

Uma issue boa reduz o espaço entre “eu queria isso” e “foi isso que foi feito”.

O que ficou dessa fase

A transformação da issue em contrato deixou algumas regras fortes.

Primeira: se está ambíguo, não implemente. Volte para documentar.

Segunda: se é importante para o fluxo, não esconda no meio de uma frase solta.

Terceira: metadado é para máquina; contexto é para humano e agente entenderem a história.

Quarta: critério de aceite precisa ser verificável, não decorativo.

Quinta: log é memória do trabalho, não rodapé cerimonial.

Foi aqui que o YODA começou a ficar menos “um monte de instruções” e mais um sistema de execução.

Mas uma coisa ainda estava pendente.

Mesmo com issue boa, ainda havia regras demais dependendo de interpretação.

Selecionar próxima issue.

Registrar log.

Atualizar status.

Avançar fase.

Tudo isso podia estar escrito.

Mas, se era tão importante assim, talvez não devesse depender só do prompt.
