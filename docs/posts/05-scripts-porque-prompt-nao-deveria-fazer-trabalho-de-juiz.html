Scripts: porque prompt não deveria fazer trabalho de juiz

Tem regra que pode ficar no texto. Tem regra que, se for importante mesmo, precisa virar comando.

Eu tentei deixar muita coisa no combinado. O agente deveria selecionar a próxima issue, respeitar dependências, atualizar log, não inventar estado, não esquecer o dev slug. Tudo lindo.

Só que combinado em prompt é igual placa de “não corra” perto da largada. Ajuda. Mas se a pessoa estiver animada, passa batido.

Foi assim que os scripts começaram a aparecer. issue_add.py, log_add.py, todo_update.py, todo_list.py, yoda_flow_next.py. Cada um nasceu de uma regra que eu não queria discutir de novo.

Script é o amigo sem carisma que salva o projeto. Ele não interpreta clima. Ele executa contrato.

Esse post é sobre tirar responsabilidade demais do prompt e colocar no lugar certo: automação pequena, determinística e testável.

O que este post vai cobrir

  • Quando uma instrução deve virar script.
  • Como runbooks de --help viraram fonte de verdade operacional.
  • Por que determinismo é carinho com o humano do futuro.
