Post 05

Scripts: porque prompt não deveria fazer trabalho de juiz

Tem regra que pode ficar no texto. Tem regra que, se for importante mesmo, precisa virar comando.

Eu tentei deixar muita coisa no combinado.

O agente deveria selecionar a próxima issue.

Respeitar dependências.

Atualizar log.

Não inventar estado.

Não esquecer o dev slug.

Não avançar fase sem autorização.

Tudo lindo.

Na teoria.

Só que combinado em prompt é igual placa de “não corra” perto da largada.

Ajuda.

Mas se a pessoa estiver animada, passa batido.

Prompt é péssimo juiz

Prompt é ótimo para orientar.

É ótimo para explicar intenção.

É ótimo para dar contexto, tom, cuidado e prioridade.

Mas prompt não deveria ser juiz de regra operacional.

Porque regra operacional precisa ser chata.

Precisa dizer sim ou não.

Precisa validar formato.

Precisa recusar estado inválido.

Precisa não acordar criativa.

O prompt, coitado, acorda criativo todos os dias.

Você pode escrever “não modifique metadata manualmente” em letras bonitas. Mas, se o agente está no meio da tarefa e acha que consegue resolver rápido, a tentação aparece.

Uma linha ali.

Um status aqui.

Um log “só para registrar”.

E pronto: metadata drift com boa intenção.

Script é o amigo sem carisma que salva o projeto. Ele não interpreta clima. Ele executa contrato.

Quando uma regra pede comando

A pergunta que começou a guiar o YODA foi simples:

Essa regra pode ser interpretada sem muito risco?

Se sim, tudo bem ficar no texto.

Mas se a regra muda estado, escolhe próxima issue, valida dependência, registra log ou altera metadata, aí a conversa muda.

Isso precisa virar comando.

Foi assim que os scripts começaram a aparecer.

issue_add.py para criar issue.

todo_list.py para inspecionar backlog.

todo_update.py para correções semânticas e metadata.

log_add.py para registrar contexto fora do fluxo normal.

yoda_flow_next.py para resolver o próximo passo determinístico.

Cada um nasceu de uma regra que eu não queria discutir de novo.

O script não precisa ser grande

Esse ponto é importante.

Quando falo “script”, não estou falando de uma plataforma gigantesca, cheia de abstração e cerimônia.

Às vezes o melhor script é pequeno.

Feio até.

Mas claro.

Ele recebe entrada, valida, decide, escreve saída e não tenta bancar o poeta.

Esse tipo de automação parece banal até você comparar com a alternativa: pedir para um agente lembrar a mesma regra em toda conversa, em todo contexto, em todo modelo.

Não obrigado.

Se é para repetir, automatiza.

Se é para validar, automatiza.

Se é para mutar estado, automatiza com ainda mais carinho.

Regra prática: prompt explica o porquê. Issue define o contrato. Script executa a parte que não pode depender de interpretação.

Runbook no --help

Uma das decisões que eu mais gosto no YODA é tratar o --help como runbook.

Não só aquele help seco, que lista flags e desaparece correndo.

Help de operador.

Para que serve.

Quando usar.

Se muta estado.

Qual é o próximo passo.

Isso é especialmente útil com agente.

Porque o agente não precisa decorar todos os detalhes operacionais do script. Ele precisa saber que, antes de usar, deve consultar o runbook que o próprio script imprime.

O script vira uma fonte de verdade viva.

E, se o comportamento muda, o runbook muda junto.

Determinismo é carinho

Existe uma frase que parece contraditória, mas ficou cada vez mais verdadeira no projeto:

Determinismo é carinho.

Não carinho fofo.

Carinho operacional.

É você impedir que o humano do futuro precise reconstruir a intenção olhando para três arquivos, uma conversa perdida e um commit com mensagem misteriosa.

É você impedir que o agente escolha uma issue diferente porque interpretou prioridade de outro jeito.

É você impedir que um dry-run mude arquivo.

É você garantir que um bloqueio não faça mutação escondida.

É você dizer: se isto aconteceu, aconteceu por este caminho.

Chato?

Sim.

Maravilhoso?

Também.

O que os scripts tiraram das costas do agente

Os scripts não tornaram o agente menos importante.

Pelo contrário.

Tiraram dele um tipo de responsabilidade que ele não deveria carregar.

O agente ainda estuda.

Ainda documenta.

Ainda implementa.

Ainda avalia.

Mas ele não precisa inventar a regra de seleção.

Não precisa decidir sozinho como escrever metadata.

Não precisa lembrar o formato do log por inspiração.

Não precisa ser juiz, escrivão, auditor e executor ao mesmo tempo.

Isso melhora o trabalho dele.

E melhora muito a minha paciência.

Quanto mais importante a regra, menos ela deveria depender de memória de conversa.

O que ficou dessa fase

A fase dos scripts consolidou algumas idéias fortes.

Primeira: scripts são obrigatórios para mudanças de metadata quando existem.

Segunda: comando que muda estado precisa validar antes de escrever.

Terceira: ajuda de CLI precisa orientar operação, não só listar flag.

Quarta: dry-run não pode ser “quase dry-run”. Ou não muta, ou não merece o nome.

Quinta: o agente deve seguir o script, e não competir com ele.

Foi aqui que o YODA começou a sair do campo da orientação e entrar no campo da orquestração.

Mas ainda faltava uma peça importante.

Como transformar uma demanda humana bagunçada em uma issue pronta para esse fluxo todo?
