YODA FrameworkVoltar para a série
Post 02

O bootstrap do bootstrap

A fase mais honesta do projeto: o framework ainda não existia, mas já precisava ajudar a criar o próprio framework.

Tem uma ironia bonita em usar um processo para construir o processo.

Bonita depois.

Na hora, parecia mais uma daquelas idéias que começam com “vai ser rápido”.

Depois você pisca e está discutindo schema, TODO, log, formato de issue, regra de seleção, slug imutável, timezone, validação mínima e o motivo pelo qual um agente não deveria sair editando arquivo só porque o nome parece familiar.

Pois é.

O YODA começou assim: tentando organizar a própria bagunça enquanto ainda descobria qual era a bagunça.

O framework que ainda não existia

No começo, não havia fluxo redondo.

Não havia pacote.

Não havia instalador.

Não havia yoda_flow_next.py apontando o próximo passo com aquela calma irritante de quem sabe exatamente o que fazer.

Havia specs.

Havia um TODO meio artesanal.

Havia decisões sendo tomadas e registradas no calor da corrida.

E havia uma fé moderada de que, se eu escrevesse o suficiente, em algum momento aquilo deixaria de ser improviso e viraria método.

Essa é a parte engraçada: o YODA nasceu defendendo documentação como fonte de verdade, mas a primeira fonte de verdade dele também estava em construção.

Meta-implementação é um nome bonito para isso.

Na prática, era mais ou menos como montar a bicicleta enquanto descia a ladeira.

O bootstrap foi o YODA de chinelo: ainda sem uniforme, mas já tentando ensinar o agente a não atravessar a rua sem olhar.

Primeiro a placa, depois o guardrail

O histórico do projeto mostra bem essa fase.

Primeiro vieram as specs, templates e o meta TODO. Depois vieram logs e regras de pendência. Depois a tradução das specs para inglês. Depois os arquivos de entrada do agente. Depois estrutura mínima. Depois interface dos scripts. Depois contrato de CLI.

Era uma sequência de pequenas placas:

Leia por aqui.

Nomeie assim.

Registre desse jeito.

Não pule dependência.

Não invente caminho.

Não confunda issue com lembrete.

No começo, muita coisa ainda era orientação escrita. O agente precisava obedecer porque estava no documento. Só que, como vimos no post anterior, documento sozinho ajuda, mas não garante.

Então as placas começaram a pedir guardrail.

Se a próxima issue deve ser escolhida por uma regra determinística, isso precisa virar script.

Se o log precisa ter formato, isso precisa virar script.

Se o status muda de forma controlada, isso precisa virar script.

O bootstrap foi o período em que essas dores foram ficando claras uma por uma.

Resumo técnico sem glamour: antes de automatizar o processo, foi preciso descobrir quais partes do processo eram importantes o bastante para não dependerem de memória, boa vontade ou interpretação.

A fase dos artefatos estranhos

Todo projeto que amadurece deixa umas conchas na praia.

No YODA, essas conchas estão em bootstrap-legacy/.

Ali ficaram TODOs antigos, logs em Markdown, logs em YAML, issues com formatos que já não são o caminho oficial, nomes que mudaram, campos que depois morreram e pequenas evidências de que a primeira versão da idéia ainda estava procurando o próprio corpo.

Dava para apagar?

Dava.

Mas apagar teria sido uma mentira elegante.

Porque aqueles arquivos contam uma coisa importante: o framework não nasceu pronto. Ele foi sendo descoberto. E algumas regras só ficaram óbvias depois de serem quebradas, ajustadas, repetidas e finalmente transformadas em contrato.

Por isso o bootstrap-legacy/ ficou.

Não como parte do fluxo atual.

Mas como fóssil útil.

Aquela gaveta que você não abre todo dia, mas que prova que a história aconteceu.

O agente também precisava de entrada

Uma das primeiras dores foi ridiculamente básica: por onde o agente começa?

Para um humano, o README costuma ser suficiente para entender a proposta geral. Para um agente, principalmente quando ele chega com contexto zero, “suficiente” é uma palavra perigosa.

Ele precisa de ordem de leitura.

Precisa saber qual arquivo tem intenção do repositório.

Precisa saber onde estão as specs.

Precisa saber qual issue manda naquele momento.

Foi assim que arquivos como REPO_INTENT.md e yoda/yoda.md começaram a ganhar importância. O projeto precisava dizer para o agente: entra por aqui, lê isso primeiro, não finja que entendeu antes da hora.

Essa preocupação parece pequena.

Não é.

Sem entrada clara, cada sessão vira um recomeço. E recomeço demais é onde mora o retrabalho.

Quando o improviso virou decisão

O momento decisivo do bootstrap não foi um commit específico.

Foi uma mudança de postura.

No começo, muita coisa era “por enquanto”.

TODO em Markdown por enquanto.

Logs manuais por enquanto.

Regras descritas em texto por enquanto.

Scripts incompletos por enquanto.

Mas todo “por enquanto” tem prazo de validade. Se ele funciona, vira contrato. Se atrapalha, vira legado. Se está no meio, vira dívida.

O bootstrap foi justamente essa triagem.

O que fica?

O que muda?

O que precisa ser automatizado?

O que era só muleta da fase inicial?

Quando os primeiros scripts começaram a aparecer, a história mudou de tom. issue_add.py, log_add.py, todo_update.py. Pequenos, específicos, ainda longe do framework final, mas já carregando uma idéia central:

Se uma regra é importante, ela não pode morar só na cabeça do agente.

O fim do bootstrap não foi o fim da bagunça. Foi só o momento em que a bagunça ganhou endereço fixo.

Por que preservar o legado

Eu gosto que o projeto tenha mantido o bootstrap-legacy/.

Não porque ele seja bonito.

Ele não é.

Mas porque ele é honesto.

Frameworks costumam ser apresentados como se tivessem nascido de uma visão clara, polida, com uma arquitetura inevitável. A realidade costuma ser mais suada: tentativa, ajuste, arrependimento, rename, campo que parecia bom, script que faltava, regra que só ficou óbvia depois do terceiro tropeço.

O legado mostra isso.

Mostra que o YODA não veio de uma grande epifania. Veio de fricção acumulada.

E fricção acumulada, quando bem registrada, vira design.

O que ficou dessa fase

O bootstrap deixou algumas lições que continuam no YODA atual.

Primeira: agente precisa de entrada explícita.

Segunda: documentação só é fonte de verdade se o processo aponta para ela sem ambiguidade.

Terceira: toda regra repetida muitas vezes está pedindo um script.

Quarta: legado não precisa mandar no presente, mas pode explicar o passado.

Quinta: criar o framework usando o próprio framework é uma péssima idéia ótima.

Péssima porque dói.

Ótima porque revela rapidamente onde o processo é frágil.

No fim, o bootstrap foi isso: a fase em que o YODA ainda não tinha maturidade para resolver todos os problemas, mas já tinha coragem suficiente para registrar os problemas certos.

E isso foi o bastante para passar para a próxima pergunta.

Se o framework vai depender de agentes, como garantir que um agente com contexto zero entre no repositório pelo lugar certo?

Pois é.

No próximo post, um agente com contexto zero entra num repositório.
