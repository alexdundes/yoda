Scripts: porque prompt não deveria fazer trabalho de juiz

Quinta publicação da série sobre a odisséia de criação do YODA Framework, pronta para copiar e publicar no LinkedIn.

Selecione o texto abaixo e cole no LinkedIn. A imagem acima pode ser usada como capa do post.

Este é o quinto texto da série sobre a odisséia de criação do YODA Framework.

No post anterior, falei sobre o momento em que a issue deixa de ser lembrete e vira contrato.

Uma issue boa não é só um título com duas frases.

Ela precisa dizer contexto, objetivo, escopo, fora de escopo, critérios de aceite, riscos, dependências e logs.

Não para burocratizar.

Para matar a telepatia.

Mas aí apareceu outra dor.

Mesmo com uma issue boa, ainda havia regras demais dependendo de interpretação.

Selecionar próxima issue.

Registrar log.

Atualizar status.

Avançar fase.

Se tudo isso era importante, talvez não devesse depender só do prompt.

Pois é.

Chegou a hora dos scripts.

Prompt é péssimo juiz

Prompt é ótimo para orientar.

É ótimo para explicar intenção.

É ótimo para dar contexto, tom, cuidado e prioridade.

Mas prompt não deveria ser juiz de regra operacional.

Porque regra operacional precisa ser chata.

Precisa dizer sim ou não.

Precisa validar formato.

Precisa recusar estado inválido.

Precisa não acordar criativa.

O prompt, coitado, acorda criativo todos os dias.

Você pode escrever “não modifique metadata manualmente” em letras bonitas.

Mas, se o agente está no meio da tarefa e acha que consegue resolver rápido, a tentação aparece.

Uma linha ali.

Um status aqui.

Um log “só para registrar”.

E pronto:

metadata drift com boa intenção.

Quando uma regra pede comando

A pergunta que começou a guiar o YODA foi simples:

Essa regra pode ser interpretada sem muito risco?

Se sim, tudo bem ficar no texto.

Mas se a regra muda estado, escolhe próxima issue, valida dependência, registra log ou altera metadata, aí a conversa muda.

Isso precisa virar comando.

Foi assim que os scripts começaram a aparecer.

issue_add para criar issue.

todo_list para inspecionar backlog.

todo_update para correções semânticas e metadata.

log_add para registrar contexto fora do fluxo normal.

yoda_flow_next para resolver o próximo passo determinístico.

Cada um nasceu de uma regra que eu não queria discutir de novo.

O script não precisa ser grande

Quando falo “script”, não estou falando de uma plataforma gigantesca, cheia de abstração e cerimônia.

Às vezes o melhor script é pequeno.

Feio até.

Mas claro.

Ele recebe entrada, valida, decide, escreve saída e não tenta bancar o poeta.

Esse tipo de automação parece banal até você comparar com a alternativa:

Pedir para um agente lembrar a mesma regra em toda conversa, em todo contexto, em todo modelo.

Não obrigado.

Se é para repetir, automatiza.

Se é para validar, automatiza.

Se é para mutar estado, automatiza com ainda mais carinho.

Runbook no help

Uma das decisões que eu mais gosto no YODA é tratar o help dos scripts como runbook.

Não só aquele help seco, que lista flags e desaparece correndo.

Help de operador.

Para que serve.

Quando usar.

Se muta estado.

Qual é o próximo passo.

Isso é especialmente útil com agente.

Porque o agente não precisa decorar todos os detalhes operacionais do script.

Ele precisa saber que, antes de usar, deve consultar o runbook que o próprio script imprime.

O script vira uma fonte de verdade viva.

Determinismo é carinho

Existe uma frase que parece contraditória, mas ficou cada vez mais verdadeira no projeto:

Determinismo é carinho.

Não carinho fofo.

Carinho operacional.

É você impedir que o humano do futuro precise reconstruir a intenção olhando para três arquivos, uma conversa perdida e um commit com mensagem misteriosa.

É você impedir que o agente escolha uma issue diferente porque interpretou prioridade de outro jeito.

É você impedir que um dry-run mude arquivo.

É você garantir que um bloqueio não faça mutação escondida.

É você dizer:

Se isto aconteceu, aconteceu por este caminho.

O que ficou dessa fase

Os scripts não tornaram o agente menos importante.

Pelo contrário.

Tiraram dele um tipo de responsabilidade que ele não deveria carregar.

O agente ainda estuda.

Ainda documenta.

Ainda implementa.

Ainda avalia.

Mas ele não precisa ser juiz, escrivão, auditor e executor ao mesmo tempo.

Isso melhora o trabalho dele.

E melhora muito a minha paciência.

Quanto mais importante a regra, menos ela deveria depender de memória de conversa.
