Décima segunda publicação da série sobre a odisséia de criação do YODA Framework, pronta para copiar e publicar no LinkedIn.

Selecione o texto abaixo e cole no LinkedIn. A imagem acima pode ser usada como capa do post.

Este é o décimo segundo texto da série sobre a odisséia de criação do YODA Framework.

E é o fechamento desta primeira temporada.

No post anterior, falei sobre o Prep Flow.

A idéia era simples e um pouco contraintuitiva:

às vezes o melhor que o agente pode fazer hoje é não implementar nada.

Preparar.

Estudar.

Documentar.

Deixar a issue menos nebulosa.

E parar.

Sem “só uma coisinha”.

Sem diff de brinde.

Sem transformar contexto fresco em ansiedade operacional.

Hoje eu fecho o ciclo inteiro.

Era só para ler o markdown

O YODA começou com uma frase pequena:

eu só queria que o agente lesse o markdown.

Era uma vontade honesta.

Quase humilde.

Daquelas que você fala e acredita que vai resolver com dois arquivos, uma convenção e talvez um README bem-intencionado.

Depois você pisca.

Tem issue com front matter.

Tem script decidindo fase.

Tem Intake.

Tem Flow.

Tem Prep Flow.

Tem pacote.

Tem install.

Tem update.

Tem log.

Tem schema.

Tem changelog.

E tem você olhando para a tela com aquela cara clássica de:

“como foi que eu vim parar aqui?”

A lição principal

A odisséia do YODA foi descobrir que pedir para o agente “seguir o contexto” não é processo.

É desejo.

Contexto solto não governa trabalho.

Ele ajuda.

Mas não garante.

Para virar processo, contexto precisa ter lugar.

Ordem.

Entrada.

Saída.

Critério.

Autorização.

Memória.

Foi aí que Markdown deixou de ser enfeite e começou a virar contrato.

Agente bom também precisa de trilho

Existe uma tentação forte de achar que modelo melhor resolve tudo.

Modelo melhor ajuda.

Prompt melhor ajuda.

Ferramenta melhor ajuda.

Mas nada disso substitui contrato.

Um agente ótimo ainda pode escolher a issue errada.

Pode implementar antes de documentar.

Pode confundir intenção com autorização.

Pode estar tecnicamente certo e operacionalmente fora do combinado.

O YODA nasceu dessa desconfiança saudável.

Não uma desconfiança cínica.

Uma desconfiança de engenharia.

Aquela que pergunta:

onde isso quebra quando eu estiver cansado?

Onde isso quebra quando o contexto estiver grande?

Onde isso quebra quando o agente estiver confiante demais?

Script é memória com dentes

Uma das viradas foi tirar regra importante do prompt e colocar em script.

Prompt é ótimo para intenção.

Script é melhor para regra.

Script não acorda inspirado.

Script não interpreta “talvez” como “pode seguir”.

Script não esquece que uma issue bloqueada não deveria ser selecionada.

Script é chato.

E por isso mesmo é útil.

O agente interpreta.

O script arbitra.

O que o YODA virou

Depois de toda essa volta, eu não diria que o YODA é só um framework para “fazer IA programar”.

Isso seria pequeno demais.

O YODA é uma tentativa de organizar a relação entre humano, agente e repositório.

O humano traz intenção.

O repositório guarda memória.

O agente executa trabalho.

O script segura o contrato.

Quando uma dessas partes tenta fazer o trabalho da outra, começa a confusão.

Humano tentando lembrar tudo de cabeça.

Agente tentando adivinhar prioridade.

Markdown virando enfeite.

Script virando sugestão.

Issue virando lembrete solto.

O YODA tenta colocar cada coisa no seu lugar.

Não perfeitamente.

Mas melhor do que “vai lá e vê o que dá”.

Vamo, vamo, vamo

No fim, a frase ficou mais engraçada do que planejada.

Vamo, vamo, vamo.

Não como pressa.

Como ritmo.

Vamo estudar antes.

Vamo documentar direito.

Vamo implementar dentro do combinado.

Vamo avaliar sem fingir que deu certo.

Vamo remover o que atrapalha.

Vamo empacotar o que precisa sair de casa.

Vamo aceitar que agente bom não elimina processo.

Ele torna processo bom ainda mais valioso.

Era só para o agente ler o markdown.

Virou uma odisséia.

Como quase tudo que começa com “vai ser rápido”.

Fecho essa primeira temporada por aqui. A próxima pergunta é a que realmente interessa: como esse processo se comporta em projetos reais, com prazos reais, dívida técnica real e humanos reais tentando não enlouquecer no meio do caminho?
