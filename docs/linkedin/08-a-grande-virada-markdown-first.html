A grande virada Markdown-first

Oitava publicação da série sobre a odisséia de criação do YODA Framework, pronta para copiar e publicar no LinkedIn.

Selecione o texto abaixo e cole no LinkedIn. A imagem acima pode ser usada como capa do post.

Este é o oitavo texto da série sobre a odisséia de criação do YODA Framework.

No post anterior, falei sobre o YODA Flow.

Study.

Document.

Implement.

Evaluate.

Quatro fases para fazer o agente desacelerar antes de sair codando como quem viu uma porta aberta.

O Flow colocou disciplina na execução.

Mas, enquanto o fluxo amadurecia, outra coisa começou a incomodar.

O formato.

Porque o YODA nasceu com YAML no nome.

YAML-Oriented Documentation & Agents.

Bonito.

Organizado.

Com cara de arquitetura.

E, como quase tudo que tem cara de arquitetura cedo demais, começou a cobrar aluguel.

YAML ajudou

Para ser justo: YAML ajudou.

No começo, fazia sentido.

YAML para TODO.

YAML para log.

YAML para metadado.

YAML para dar aquela sensação de que a máquina finalmente teria uma planilha espiritual para obedecer.

YAML é bom para dado estruturado.

Campo.

Valor.

Lista.

Flag.

Coisa que script lê sem drama.

O problema começou quando YAML tentou virar diário.

Issue não é formulário

Uma issue não é só um formulário.

Ela tem contexto.

Tem decisão.

Tem hesitação.

Tem “isso parece simples, mas cuidado com aquele detalhe”.

Tem a frase atravessada que explica por que uma alternativa foi descartada.

Tem o medo pequeno que não cabe em boolean.

YAML não gosta disso.

Ele até aceita.

Mas aceita com aquela cara de quem está segurando um móvel no elevador.

Você começa a escrever texto longo dentro de campo.

Depois quebra linha.

Depois indenta.

Depois um caractere inocente vira incidente diplomático.

E aí você percebe:

talvez eu esteja usando o armário como sala de estar.

A duplicação começou a feder

No modelo inicial, havia estado em mais de um lugar.

Um TODO YAML dizia uma coisa.

A issue Markdown dizia outra.

O log YAML guardava histórico.

O markdown tentava contar a história.

O agente precisava montar o quebra-cabeça antes de entender o que estava acontecendo.

Isso funciona por um tempo.

Como muita coisa funciona por um tempo em software.

Esse é o perigo.

Funciona o suficiente para você confiar.

Depois quebra no pior momento, com uma divergência pequena e educada.

Um status aqui.

Uma fase ali.

Um log que ficou no arquivo antigo.

Uma dependência certa em um lugar e ausente no outro.

Pronto.

Agora você não tem fonte de verdade.

Tem fonte de debate.

Markdown tomou o centro

A virada foi simples de explicar e trabalhosa de assumir:

a issue Markdown virou a fonte operacional de verdade.

Metadados no front matter.

Contexto no corpo.

Critérios de aceite no corpo.

Notas de implementação no corpo.

Resultado no corpo.

Flow log no corpo.

Um arquivo só contando a história inteira.

O YAML não morreu.

Ele só parou de querer ser protagonista.

Virou etiqueta.

Markdown virou documento.

Menos duplicação, menos teatro

Outra decisão pequena, mas libertadora:

o ID passou a ser derivado do nome do arquivo.

Se o arquivo se chama yoda-0049-alguma-coisa.md, o ID é yoda-0049.

Fim.

Não precisa duplicar.

Não precisa reconciliar.

Não precisa abrir investigação porque o filename diz uma coisa e o front matter diz outra.

O log também mudou de casa.

Foi para dentro da issue.

Porque a pergunta sempre era:

“O que aconteceu com essa issue?”

E a resposta não deveria exigir caça ao tesouro.

A virada 0.3.0

Na prática, isso virou a transição do modelo 0.3.0.

TODO YAML legado virou metadado em issue Markdown.

Logs YAML foram consolidados no Flow log.

Scripts passaram a operar a partir de yoda/project/issues/*.md.

Com filtro por dev.

Com ID derivado do filename.

Com validação de schema.

Com erro claro quando algo estava errado.

Foi a hora em que a preferência estética virou contrato operacional.

Markdown venceu porque é legível.

Porque aceita contexto.

Porque carrega decisão.

Porque conversa bem com git.

Porque o humano realmente lê.

E porque o agente opera melhor quando não precisa montar a história a partir de pedaços espalhados.

YAML continuou útil.

Só perdeu a mania de querer narrar a novela.

No próximo texto, a dor muda de lugar: depois de organizar o fluxo, chegou a hora de empacotar uma teimosia. Porque criar um framework no próprio repositório é uma coisa. Fazer ele caber num pacote instalável é outra história.
