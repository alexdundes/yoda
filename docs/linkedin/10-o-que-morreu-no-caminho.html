LinkedIn - Post 10

O que morreu no caminho

Décima publicação da série sobre a odisséia de criação do YODA Framework, pronta para copiar e publicar no LinkedIn.

Selecione o texto abaixo e cole no LinkedIn. A imagem acima pode ser usada como capa do post.

Este é o décimo texto da série sobre a odisséia de criação do YODA Framework.

No post anterior, falei sobre empacotar uma teimosia.

Ou seja: tirar o YODA do conforto do meu próprio repositório e transformar aquilo em pacote distribuível.

Com manifest.

Changelog.

latest.json.

Checksum.

Instalador.

Update.

Rollback.

Foi ali que o projeto começou a se comportar como framework de verdade.

Mas framework de verdade não amadurece só adicionando coisa.

Ele amadurece removendo também.

E esse post é sobre isso.

O que morreu no caminho.

Remover parece derrota

Criar conceito é gostoso.

Dá sensação de avanço.

Você nomeia uma coisa e, por alguns minutos, parece que resolveu um problema.

Às vezes resolveu mesmo.

Às vezes só colocou um crachá no problema e deu uma cadeira para ele na reunião.

Remover é mais difícil.

Remover parece admitir erro.

E ninguém gosta de olhar para uma issue antiga e pensar:

“olha que bela complicação eu inventei com convicção”.

Mas remover nem sempre é derrota.

Muitas vezes é o projeto ficando mais honesto.

O fim do lightweight

O lightweight nasceu com uma intenção boa.

Ter um caminho mais leve.

Uma variação menor.

Um modo menos cerimonial.

Quem nunca quis isso?

O problema é que um modo alternativo nunca é gratuito.

Ele aparece no schema.

Nos scripts.

Nos templates.

Nos testes.

Na cabeça do agente.

Na cabeça do humano.

E cria uma pergunta que não ajuda:

“isso aqui é flow normal ou lightweight?”

No uso real, não pagava a própria manutenção.

Então saiu.

Tags e agent

Tags também pareciam uma boa idéia.

Tags sempre parecem.

Elas prometem organização sem compromisso.

Até ninguém saber qual tag usar.

Ou duas tags significarem quase a mesma coisa.

Ou o filtro existir e ninguém filtrar.

No YODA, tags viraram schema a mais para valor de menos.

Morreram também.

O campo agent foi parecido.

Ele soava correto.

Afinal, estamos falando de agentes.

Mas o YODA não precisava desse campo para selecionar issue, avançar fase ou validar aceite.

Era semanticamente bonito.

Operacionalmente dispensável.

Foi embora.

Duplicação disfarçada de segurança

Algumas mortes vieram da mesma doença:

duplicação.

Entry points no YAML e no Markdown.

ID no filename e no front matter.

Dependencies no front matter e no corpo.

Parece seguro repetir.

Mas repetição de estado quase sempre vira convite para divergência.

Um lugar atualiza.

O outro não.

E pronto.

Você não tem fonte de verdade.

Tem duas testemunhas discordando.

O YODA foi limpando isso.

Entry points ficam no Markdown.

ID vem do filename.

Dependência operacional fica em depends_on.

Um dado.

Um lugar.

Menos reconciliação depois.

Template sem fingir intenção

Template com default demais é uma mentira confortável.

Ele parece pronto.

Mas parte daquilo é só preenchimento automático fingindo intenção.

No YODA, o template chegou a carregar metadata operacional demais.

Aí a responsabilidade ficava espalhada.

O template define?

O script define?

O update preserva?

O init reconcilia?

Quando muita gente manda no mesmo campo, o campo vira condomínio.

E condomínio tem reunião.

A solução foi tirar defaults operacionais do template.

Template virou estrutura.

Scripts viraram fonte de preenchimento.

Init não é dono da casa

Essa foi uma remoção mais recente e importante.

O init.py já foi mais intrusivo.

Ele mexia em arquivos de agente e intent na raiz do projeto host.

Isso parecia útil para onboarding.

Até você lembrar que o projeto host tem dono.

Ele pode ter seu próprio AGENTS.md.

Seu próprio GEMINI.md.

Seu próprio CLAUDE.md.

Seu próprio contexto.

O YODA não deveria entrar e reorganizar a geladeira.

A solução foi deixar as instruções autocontidas dentro de yoda/.

O host aponta para elas se quiser.

Mas o YODA não invade a raiz.

O padrão

O padrão se repetiu várias vezes.

Uma idéia parecia útil.

Entrava no schema.

Ganhava flag.

Ganhava teste.

Ganhava documentação.

Depois o uso real mostrava que ela não pagava a própria manutenção.

O YODA ficou melhor quando parou de confundir completude com quantidade de campos.

Menos metadata.

Menos duplicação.

Menos modos alternativos.

Menos invasão no projeto host.

Mais fonte única.

Mais Markdown onde a história importa.

Mais script onde regra precisa ser chata.

Adicionar mostra imaginação.

Remover mostra critério.

No próximo texto, a história muda de direção: depois de remover, limpar e estabilizar, apareceu a necessidade de preparar antes de executar. Às vezes a melhor coisa que o agente pode fazer hoje é não implementar nada.
