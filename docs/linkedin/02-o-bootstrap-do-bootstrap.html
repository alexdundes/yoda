Este é o segundo texto da série sobre a odisséia de criação do YODA Framework.

No primeiro post, eu contei de onde veio a dor inicial:

Eu só queria que o agente lesse o markdown.

Só isso.

Escrever uma spec, entregar para o agente, ele implementar direitinho e seguir a vida.

Mas aí apareceu o problema do “entendi”.

O agente dizia que tinha entendido, implementava algo plausível, e eu descobria no diff que uma parte importante do combinado tinha virado interpretação.

Daí veio a primeira tese do YODA:

Antes do código, o combinado.

Só que essa frase bonita trouxe um problema novo.

Como construir um framework para organizar agentes se o próprio framework ainda não existia?

Pois é.

Bem-vindo ao bootstrap do bootstrap.

O framework que ainda não existia

Tem uma ironia bonita em usar um processo para construir o processo.

Bonita depois.

Na hora, parecia mais uma daquelas idéias que começam com “vai ser rápido”.

Depois você pisca e está discutindo schema, TODO, log, formato de issue, regra de seleção, slug imutável, timezone, validação mínima e o motivo pelo qual um agente não deveria sair editando arquivo só porque o nome parece familiar.

No começo, não havia fluxo redondo.

Não havia pacote.

Não havia instalador.

Não havia script apontando o próximo passo com aquela calma irritante de quem sabe exatamente o que fazer.

Havia specs.

Havia um TODO meio artesanal.

Havia decisões sendo tomadas e registradas no calor da corrida.

E havia uma fé moderada de que, se eu escrevesse o suficiente, em algum momento aquilo deixaria de ser improviso e viraria método.

Meta-implementação é um nome bonito para isso.

Na prática, era mais ou menos como montar a bicicleta enquanto descia a ladeira.

Primeiro a placa, depois o guardrail

O bootstrap foi o período das primeiras placas.

Leia por aqui.

Nomeie assim.

Registre desse jeito.

Não pule dependência.

Não invente caminho.

Não confunda issue com lembrete.

No começo, muita coisa ainda era orientação escrita.

O agente precisava obedecer porque estava no documento.

Só que documento sozinho ajuda, mas não garante.

Então as placas começaram a pedir guardrail.

Se a próxima issue deve ser escolhida por uma regra determinística, isso precisa virar script.

Se o log precisa ter formato, isso precisa virar script.

Se o status muda de forma controlada, isso precisa virar script.

Foi assim que o YODA começou a sair do “agente, por favor leia isso” para “agente, execute este comando e siga o runbook”.

A gaveta histórica

Todo projeto que amadurece deixa umas conchas na praia.

No YODA, essas conchas ficaram em uma pasta chamada bootstrap-legacy.

Ali ficaram TODOs antigos, logs em Markdown, logs em YAML, issues com formatos que já não são o caminho oficial, nomes que mudaram, campos que depois morreram e pequenas evidências de que a primeira versão da idéia ainda estava procurando o próprio corpo.

Dava para apagar?

Dava.

Mas apagar teria sido uma mentira elegante.

Porque aqueles arquivos contam uma coisa importante:

O framework não nasceu pronto.

Ele foi sendo descoberto.

E algumas regras só ficaram óbvias depois de serem quebradas, ajustadas, repetidas e finalmente transformadas em contrato.

O que ficou dessa fase

O bootstrap deixou algumas lições que continuam no YODA atual.

Primeira: agente precisa de entrada explícita.

Segunda: documentação só é fonte de verdade se o processo aponta para ela sem ambiguidade.

Terceira: toda regra repetida muitas vezes está pedindo um script.

Quarta: legado não precisa mandar no presente, mas pode explicar o passado.

Quinta: criar o framework usando o próprio framework é uma péssima idéia ótima.

Péssima porque dói.

Ótima porque revela rapidamente onde o processo é frágil.

No fim, o bootstrap foi isso:

A fase em que o YODA ainda não tinha maturidade para resolver todos os problemas, mas já tinha coragem suficiente para registrar os problemas certos.
