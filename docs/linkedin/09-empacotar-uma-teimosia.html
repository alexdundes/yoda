Selecione o texto abaixo e cole no LinkedIn. A imagem acima pode ser usada como capa do post.

Este é o nono texto da série sobre a odisséia de criação do YODA Framework.

No post anterior, falei sobre a grande virada Markdown-first.

O YODA nasceu com YAML no nome.

Mas, em algum momento, ficou claro que YAML era ótimo para metadado e péssimo para contar história.

A issue Markdown virou fonte operacional de verdade.

O YAML ficou como etiqueta.

O Markdown ficou como documento.

Bonito.

Organizado.

Quase maduro.

Aí veio a pergunta que estraga qualquer conforto:

tá, mas como isso sai do meu repositório?

Pois é.

Uma coisa é funcionar no meu projeto.

Outra é virar pacote.

O pacote é onde a fantasia acaba

Enquanto o YODA estava só no próprio repositório, muita coisa funcionava por proximidade.

O script estava ali.

A spec estava ali.

O histórico estava ali.

A bagunça também estava ali, mas pelo menos era uma bagunça conhecida.

É como morar sozinho.

A cadeira no meio da sala não é problema.

É layout.

Mas, quando o framework precisa entrar em outro projeto, a história muda.

Ele precisa se comportar como visita educada.

Não pode quebrar a casa dos outros.

Não pode levar bastidor demais.

Não pode apagar dado local.

Não pode sair mexendo em README da raiz como se fosse dono do lugar.

Empacotar é quando a fantasia acaba.

O que entra e o que fica fora

A primeira decisão foi simples:

o pacote não podia ser o repositório inteiro.

Não fazia sentido levar project/specs para todo projeto consumidor.

Não fazia sentido levar bootstrap histórico.

Não fazia sentido levar testes, cache, arquivos temporários e lembranças emocionais.

O pacote precisava carregar só o que o projeto host precisava para usar o YODA.

Manual embutido.

Entrypoints locais para agentes.

Scripts.

Templates.

Licença.

README curto.

Changelog.

Manifest.

Só.

O resto é bastidor.

E bastidor não deveria cair na mochila do usuário.

package.py: o adulto responsável

O package.py virou a fonte da verdade do empacotamento.

Porque empacotar na mão é uma forma criativa de gerar releases com personalidade própria.

Uma inclui arquivo demais.

Outra esquece um script.

Outra sai com spec interna.

Outra tem changelog escrito no improviso.

E depois, claro, ninguém arruma.

Então o pacote virou comando.

Determinístico.

Com lista de inclusão.

Com lista de exclusão.

Com tar.gz.

Com versão.

Com build metadata.

Com changelog estruturado.

Com latest.json.

Com checksum.

Menos “acho que gerei certo”.

Mais “o comando sabe o contrato”.

Checksum e latest.json

O latest.json virou o cartão de visita remoto do pacote.

Ele diz a versão.

Diz o build.

Diz onde baixar.

Diz o sha256.

O instalador usa isso para baixar e verificar.

Não é uma solução mágica de segurança.

Não tenta fingir que resolveu toda a cadeia de suprimentos do planeta.

Mas já impede acidente bobo.

Baixou errado?

Corrompeu?

Hash não bateu?

Para.

Essa é uma das palavras mais bonitas da engenharia:

para.

O instalador de uma linha

Instalador de uma linha é lindo.

E meio assustador.

curl | sh é aquela coisa que todo mundo já viu, muita gente usa e quase todo mundo explica com uma tosse moral antes.

Por isso o YODA precisou ser honesto.

Existe o caminho rápido.

E existe o caminho mais responsável.

O rápido instala a última versão.

O responsável fixa uma versão.

O manual baixa, confere checksum e instala com mais controle.

Não é sobre fingir que facilidade não tem risco.

É sobre deixar o risco visível.

Update, backup e rollback

Instalar é só o primeiro encontro.

O relacionamento começa no update.

Atualizar framework embutido é delicado porque você quer trocar o motor sem jogar fora o porta-luvas.

O update precisava verificar versão.

Validar checksum.

Fazer backup.

Substituir só arquivos do framework.

Preservar dados do projeto.

Rodar init quando necessário.

E deixar uma história de volta.

Rollback não é pessimismo.

Rollback é respeito por terça-feira às 17h42.

Porque é exatamente nesse horário que uma atualização aparentemente simples decide testar sua humildade.

Release também é fluxo

No fim, empacotamento virou mais um fluxo.

Leia o histórico.

Derive changelog de mudanças reais.

Gere o pacote.

Verifique o conteúdo.

Confirme que não levou o que não devia.

Confirme que levou o que precisava.

Atualize metadata.

Entregue um artefato rastreável.

Release sem isso vira superstição com número de versão.

E superstição com número de versão ainda é superstição.

Empacotar ensinou uma coisa importante:

um framework só começa a virar framework quando consegue sair do ambiente onde nasceu.

Antes disso, ele é uma idéia local com bons argumentos.

No próximo texto, a gente visita o cemitério técnico: conceitos que pareciam bons, campos que nasceram confiantes e idéias que morreram no caminho porque o projeto ficou mais honesto.
